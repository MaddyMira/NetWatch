Guide
How to get the most out of NETWATCH
🚀 Getting Started

NETWATCH is a local network monitor — it runs on your machine and polls every device you add using ICMP ping. No agents needed on monitored hosts.

1
Add your devices — go to Devices and click + Add Device. Enter the IP (IPv4 or IPv6) and a friendly name.
2
Start monitoring — click Start Monitoring in the sidebar. The pulse dot turns green when active.
3
Watch the dashboard — each device card shows live status, latency, and last-check time. Cards flash red on first offline detection.
4
Set up alerts — go to Alert Rules to be notified when a device goes offline, latency spikes, or packet loss exceeds a threshold.
💡 Use Ctrl+K to jump to any section instantly.
📡 Monitoring Correctly

The monitor sends ICMP echo requests to each device on a configurable interval. The key settings:

Interval
10 s is a good LAN default. Use per-device overrides for critical servers (5 s) or low-priority devices (60 s) — see Poll Intervals.
Timeout
2–3 s for local network. Increase to 5 s for VPN or remote links only.
Max Retries
2 retries (3 total attempts) prevents false positives from transient hiccups.
⚠️ On Windows, run NETWATCH as Administrator for raw ICMP. On Linux: sudo setcap cap_net_raw+ep $(which python3)
⊞ Managing Devices
Groups
Tag devices by function (Servers, Printers, Cameras…). Enables colour-coded filtering on the Dashboard and visual clusters in the Topology map.
Notes
Document location, hardware model, or owner. Visible in the device table and exported reports.
Poll Interval
Per-device poll rate override. Blank = global default. See Poll Intervals.
Reorder
Use Reorder to drag devices into the display order you want — persists across restarts.
💡 Run a Discovery scan, tick the hosts you want, and click + Add Selected to bulk-add devices.
◉ Network Discovery

The Discovery scanner pings every host in a CIDR range concurrently — purely passive, ICMP only.

1
Go to Discovery. NETWATCH auto-detects your network interfaces and suggests the most likely subnet.
2
Select a subnet (or type a custom CIDR like 10.10.0.0/16) and click Start Scan.
3
Alive hosts appear in real time. Hostname resolution runs in a second pass. Uncheck it to skip on large subnets.
4
Tick the hosts you want and click + Add Selected.
⚠️ Large subnets (/16 = 65,534 hosts) take 30–60 s and generate significant ICMP traffic. Stick to /24 for routine scans.
🔔 Setting Up Alerts Effectively

Alert rules evaluate after every monitor cycle. Four rule types are available:

offline
Device has been OFFLINE continuously for at least Duration seconds. Set Duration to 30 s to absorb brief blips.
online
Device just came back online (status flip). Useful to confirm a reboot completed.
latency_gt
Latency exceeds threshold (ms) continuously. Good thresholds: 50–100 ms for LAN, 150–300 ms for WAN/VPN.
packet_loss_gt
Real rolling loss % (last 20 pings) exceeds threshold. 10% for wired, 25% for Wi-Fi.

Configure channels in Notifications first, then assign them per rule.

💡 A single "any device offline for 60 s → all channels" rule is a good baseline. Add per-device latency rules for critical servers on top.
🔧 Network Tools
Port Scanner
TCP connect scan with banner grabbing. Quick = 16 common ports. Common = all well-known ports. Full = ports 1–1024 + high ports. Not UDP — ICMP/UDP services won't appear.
Traceroute
Maps the hop-by-hop path to a target via the OS tracert/traceroute. * * * hops mean that router doesn't respond to probes — completely normal for most firewalls.
ARP Table
Reads the OS ARP cache and resolves MAC OUI to vendor. Refresh after discovery to map physical devices to IPs. Entries expire after a few minutes of inactivity.
DNS / Geo
A, PTR, MX, NS, TXT lookups. Geo uses ip-api.com (free, 45 req/min). Private IPs return "Private Network".
⏻ Wake-on-LAN

WoL sends a magic packet — a UDP broadcast with the target's MAC repeated 16 times — to wake a powered-off machine. The machine must have WoL enabled in firmware and OS.

1
Enable in BIOS/UEFI — look for "Wake on LAN", "Power On By PCI-E", or "EuP Ready" under Power Management.
2
Enable in the OS — Windows: Device Manager → NIC → Advanced → Wake on Magic Packet → Enabled. Linux: sudo ethtool -s eth0 wol g
3
Get the MAC address — use the ARP Table or run arp -a while the device is on.
4
Send the packet — go to Wake-on-LAN, enter the MAC and broadcast address, click Send Magic Packet. NETWATCH pings the target every 2 s for 60 s and reports when it comes up.
⚠️ WoL only works within the same broadcast domain. It will not cross routers unless directed broadcast forwarding is enabled — which is off by default on most equipment.
📉 Understanding Packet Loss

NETWATCH calculates real packet-loss % from the last 20 ping results in the database per device — a rolling window that reflects recent link quality.

0%
All 20 recent pings replied. Healthy.
5–15%
Intermittent — acceptable on Wi-Fi or a congested link. Watch closely.
15–50%
Degraded — bad cable, switch port, or Wi-Fi interference. Alert-worthy.
>50%
Severe — link is effectively down intermittently. Escalate immediately.
💡 The Live Charts section shows an hourly packet-loss graph per device over the last 24–48 hours.
⏱ Per-Device Poll Intervals

Every device can have its own poll interval. Set it in the device editor (pencil icon on Devices). Leave blank to inherit the global default.

Critical servers
5 s — near-real-time outage detection.
Workstations / PCs
15–30 s — notice when someone turns a machine off.
Printers / NAS
60 s — these rarely change state; save polling overhead.
IoT / cameras
30–120 s — adjust to criticality.
💡 The engine ticks every 0.5 s, so the effective minimum interval is 1 s. Setting it to 0 or blank reverts to the global setting.
🌐 IPv6 Support

Enter an IPv6 address when adding a device and NETWATCH will ping it via ICMPv6 using the OS ping command.

Global unicast
2001:db8::1 — routable, like an IPv4 public address.
Link-local
fe80::1%eth0 — local segment only. Include the interface suffix on Linux.
Unique local
fd00::1 — equivalent to RFC-1918 private ranges.
⚠️ Discovery is IPv4-only. Add IPv6 devices manually or look them up via the ARP table.
💡 Port Scanner and Traceroute also work with IPv6 targets.
⟳ REST API & Prometheus

Enable the HTTP API in REST API to integrate NETWATCH with external dashboards and automation.

Bind address
127.0.0.1 = localhost only (safe default). 0.0.0.0 = all interfaces (trusted networks only).
Grafana
Point a Prometheus datasource at http://<host>:8765/metrics. Metrics include per-device uptime, latency, and packet-loss counters.
Automation
Use /api/status, /api/devices, and /api/alerts to pipe NETWATCH data into scripts or other tools.
⚠️ The API has no authentication. Use firewall rules or a reverse proxy with auth if exposing it beyond localhost.
